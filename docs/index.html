Description

For my project, I want to make an event planning application. My goal is to make an app that lets people organize events, invite people, and define other parts of a party. I would like to use keys to access events which you would get from others, item lists for BYOB events, Google Maps integration for quick access to the event location, and a chat feature that all invited guests could use.

Design

Technologies
For technologies, while I started this project in Spring, I switched and it's now primarliy PHP, SQL, and HTML. I've also successfully deployed my app on Google Cloud Services using the App Engine and Cloud SQL. I would like to integrate the API for Google Maps to show locations for events you've been invited to, or possibly even show the locations of multiple parties you've been invited to in one place.
Use Cases
For my use cases, core functionality would include:
  • Generating an event from a form
  • Accessing a page for details about a specific party
After those core tasks are completed, more beneficial functionality would be added based on priority. Throughout the timeline of the project, I want to include:
  • An item list for people to bring items
  • Google Maps integration for quick access to directions
This list contains functionality that I want to implement if time allows. Obviously, the largest portion of this will be making sure the MySQL integrates well and the schema for the database makes sense given the context of the application.

Justification

Novelty
I'm not aiming to create something amazing with this application, but I do think this idea has enough unique features to make it something that fits a niche market of people. There are alternatives to individual features; you can chat on Messenger, text your address to people, or create a Facebook event for your party, but I think tying these specific features together allows for a much nicer way to get people together and organize an event.
I see a few issues with what currently exists. They aren't cohesive, they're fragmented portions of an app that don't fully support an event-centered use case. Facebook event pages are close, but it's still focused on posts, which is not intuitive for an event. There are no lists for items on Facebook, you can't send the directions to your phone through Google Maps, you can't easily chat about the event with the other guests outside of posts unless you use Messenger, and you can't see all the events you've been invited to without digging through the rest of Facebook's endless side panel.
Complexity
The complexity of this app comes from not just the languages used (PHP, SQL), but primarily from configuring Google Cloud Services effectively. GCS needed configured in two ways. The App Engine needed to have my application uploaded to it to create instances, and the Cloud SQL needed to be created and configured to accept requests from the App Engine.
